# Agent Task: News Aggregation Bot Implementation Plan

## Goal

Build a personal AI news aggregation system with:

1. Telegram channel aggregation from a personal Telegram account.
2. Gmail newsletter/subscription aggregation via Gmail OAuth.
3. Website/RSS scraping from a configurable source list.
4. Trusted-source preprocessing and deduplication.
5. AI summarization by template.
6. Local embeddings and PostgreSQL vector search.
7. Telegram bot frontend for delivery/search/submission.
8. Optional web frontend later.
9. One-command Docker startup.

Primary stack:

- Python 3.14
- PostgreSQL + pgvector
- Local embedding model: `sentence-transformers/all-MiniLM-L6-v2`
- Telegram user access: Telethon session/auth
- Telegram bot publishing: Bot API token
- Gmail access: OAuth
- LLM access: implement real `openai-codex` OAuth, compatible with pi's OpenAI Codex auth flow
- Smart model default: `gpt-5.5`
- Cheap/simple model default: `gpt-5.4-nano`

All model/provider names must be configurable.

---

## Non-goals for MVP

Do not build a complex web frontend first.
Do not rely on embeddings alone for final story deduplication.
Do not hard-delete rejected/duplicate data; mark statuses for debugging.

## Preprocessing design

### Step 1: Extract URLs

From:

- Telegram message text/entities
- email HTML/text
- RSS item links
- scraped page links
- user-submitted Telegram messages

Store in `raw_item_links`.

### Step 2: Normalize URLs

Implement `preprocessing/urls.py`:

- remove UTM/tracking parameters
- lower-case host
- remove default ports
- remove fragments unless needed
- remove trailing slash where safe
- unwrap known redirect/tracking links from newsletters
- follow redirects with timeout
- discover canonical URL from page metadata

### Step 3: Reject forbidden domains early

Reject by rule:

- Telegram links as evidence/source links
- Facebook links
- non-English pages
- known spam/content farms
- unsupported binary files

X/Twitter rule:

- Accept `x.com` only if the account is present in `trusted_social_accounts` and the post is original/official/trusted.
- Reject random X reposts/rumors.

### Step 4: Visit and parse links

Use:

- `httpx`
- `trafilatura`
- `readability-lxml`
- `beautifulsoup4`
- optional Playwright later for JS-heavy sites

Store each fetched page as `documents`.

### Step 5: Discover trusted/root source

For each document:

1. Parse outbound links.
2. Look for official/company/source-of-truth links.
3. Follow links up to `MAX_LINK_DEPTH`.
4. Prefer sources in this order:

```text
1. official company/project blog
2. official press release
3. official documentation/changelog/GitHub release
4. official X account posting its own news
5. trusted major publication
6. trusted newsletter with citations
```

Mark:

```text
source_role = root/primary/secondary/mention/rejected
is_root_source = true/false
trust_level = primary/trusted/allowed/rejected/unknown
```

### Step 6: Language validation

Require English if `REQUIRE_ENGLISH=true`.

Use fast language detection first, then optional LLM validation if uncertain.

### Step 7: Rejection rule

If no trusted English source/document is found:

```text
raw_items.status = rejected_no_trusted_source
```

Do not summarize.
Do not hard-delete.

## Deduplication design

Deduplication happens in multiple layers.

### Dedup 1: Raw item dedup

Before processing:

- Telegram: `(source_id, telegram_message_id)`
- Gmail: `(source_id, gmail_message_id)` or Message-ID
- RSS: `(source_id, guid)`
- Web: canonical URL/hash

If duplicate:

```text
status = duplicate_raw_item
```

### Dedup 2: URL/canonical URL dedup

After URL normalization/canonicalization:

- if `canonical_url` already exists, reuse existing `document`
- if `root_source_url` already exists, attach to same candidate story/source bundle

### Dedup 3: Content hash dedup

If cleaned article text hash already exists:

- reuse document
- mark duplicate by `content_hash`

### Dedup 4: Pre-summary source bundle dedup

Before summarization, build a source bundle:

```text
root source document
+ trusted secondary documents
+ original raw item context
```

If this bundle has the same root source URL or same root document hash as an already summarized bundle:

- reuse/update existing summary if needed
- do not create duplicate summary

### Dedup 5: Embedding candidate search

After summary creation:

- embed `summary_title + summary_text`
- use pgvector to find candidate similar summaries/clusters

Important:

```text
Embeddings are candidate retrieval only, not final truth.
```

### Dedup 6: Hybrid story matching

For each candidate, decide if it is the same real-world event using:

- same root/official source URL
- same canonical URL
- same content hash
- same extracted entities
- same event type
- same product/company/person
- same approximate event date
- embedding similarity
- LLM adjudication for uncertain cases

### LLM adjudication prompt

For uncertain matches, call smart model and ask:

```text
Are these two summaries about the same real-world news event,
or only related/same-topic?

Return JSON:
{
  "same_story": true/false,
  "confidence": 0.0-1.0,
  "reason": "...",
  "shared_facts": ["..."],
  "different_facts": ["..."],
  "recommended_relation": "same_story|related|separate"
}
```

Auto-merge only if:

- deterministic same URL/root URL/content hash, or
- LLM says `same_story=true` with high confidence, and structured entities/event type do not conflict.

Otherwise:

- create `related_stories`, or
- leave separate.

## OpenAI Codex OAuth implementation

This project must implement the actual `openai-codex` OAuth flow, not a placeholder generic API-key client.

Use pi's implementation as the reference. Known pi references on this machine:

```text
/mnt/c/Users/MaSliusareva/.pi/agent/extensions/openai-dmitry.ts
~/.pi/agent/auth.json
```

The extension shows the intended API shape:

```ts
import { loginOpenAICodex, refreshOpenAICodexToken } from "@mariozechner/pi-ai/oauth";
```

and registers provider API:

```text
api = openai-codex-responses
provider = openai-codex-compatible provider
```

pi's stored `openai-codex` credentials have this shape:

```json
{
  "type": "oauth",
  "access": "...",
  "refresh": "...",
  "expires": 1234567890,
  "accountId": "..."
}
```

Do not print or commit real tokens.

### Required Python modules

Implement:

```text
app/ai/openai_codex_oauth.py
app/ai/openai_codex_client.py
scripts/init_oauth.py openai-codex
```

### OAuth behavior

The implementation must support:

1. Interactive login using a local callback server.
2. Saving credentials to `LLM_OAUTH_TOKEN_PATH`.
3. Importing existing pi credentials from `PI_AUTH_JSON_PATH` if available.
4. Refreshing access tokens with the refresh token before expiry.
5. Returning the current bearer token to the LLM client.
6. Preserving `accountId` if returned/available.

Credential file format should match pi as closely as possible:

```json
{
  "type": "oauth",
  "access": "access_token",
  "refresh": "refresh_token",
  "expires": 1234567890,
  "accountId": "optional_account_id"
}
```

### LLM client behavior

`openai_codex_client.py` should call the Codex/OpenAI Responses-compatible endpoint using:

```text
Authorization: Bearer <access token>
```

and should route calls by task type:

```text
smart -> LLM_SMART_MODEL
cheap -> LLM_CHEAP_MODEL
```

If web search is needed, use the provider's Responses/web-search capability when supported by the selected model.

### Implementation note

Before coding this part, inspect pi/node implementation further if installed locally. The first implementation can either:

- port the OAuth logic directly to Python, or
- invoke a small Node helper that uses the same `@mariozechner/pi-ai/oauth` functions, if those package internals are accessible in the runtime.

Preferred long-term result: pure Python OAuth/token refresh + Python client.

## Summarization design

Only summarize after:

- trusted English source found
- source documents parsed
- URL/content/source-bundle dedup performed

Input:

```text
root/official source text
+ trusted secondary sources
+ original raw item context
+ all trusted source links
```

Output must preserve source links.

Suggested summary template:

```text
Title:

What happened:

Why it matters:

Key facts:
- ...

Source quality:
- Root source: ...
- Secondary sources: ...

Links:
1. ...
2. ...
```

Store in `summaries` and `summary_links`.

## Acceptance criteria

MVP is complete when:

1. `docker compose up --build` starts Postgres, app, worker, and bot.
2. Gmail OAuth can be initialized and newsletters collected.
3. Telegram personal account auth can be initialized and channel posts collected.
4. Telegram bot can publish digest to owner.
5. Raw items, links, documents, summaries, embeddings, clusters are stored in Postgres.
6. Links are normalized and forbidden domains are rejected.
7. Non-English and no-trusted-source items are rejected before summarization.
8. Summaries are generated only from trusted source bundles.
9. Embeddings are local using `all-MiniLM-L6-v2`.
10. pgvector search works.
11. Story clustering does not rely on embeddings alone.
12. Source links are preserved in every delivered summary.
13. Model names and credentials are configurable via environment variables.

## Important design rules

- Preserve raw data for audit/debugging.
- Mark rejected/duplicate statuses; do not hard-delete.
- Links are first-class data, not just JSON arrays.
- Trusted/root source discovery happens before summarization.
- Embeddings are for candidate retrieval, not final dedup truth.
- Use LLM adjudication only when cheaper deterministic rules are insufficient.
- Keep individual summaries and also create combined cluster summaries.
- Separate clusters from tags.
- Keep Telegram bot owner-only by default.
